Hooks:
o essencial

Um hook é uma ordem automática: "toda vez que acontecer isto, rode aquilo". É como o Claude Code segue suas regras sozinho — sem depender de lembrar.

↓ role para estudar

01 O que é um hook

Um hook é uma ordem automática que você dá ao Claude Code: "toda vez que acontecer isto, rode aquilo". O isto é um momento (um evento); o aquilo é um comando seu — um script, um formatador, um aviso. O Claude dispara sozinho, sem você pedir de novo.

Sem hook, você repete as mesmas instruções ("formata o código", "não roda isso") a cada sessão, e torce pra elas serem seguidas. Com hook, a regra vira parte do ambiente: automática, e o mesmo comportamento toda vez.

02 Os momentos (eventos)

Cada hook se pendura num evento — um momento do trabalho do Claude. Os que mais importam pra começar:

  • PreToolUse — antes do Claude usar uma ferramenta (ex.: antes de rodar um comando no terminal).
  • PostToolUse — depois (ex.: formatar o arquivo assim que ele é editado).
  • UserPromptSubmit — quando você envia uma mensagem.
  • Stop — quando o Claude termina de responder.
  • SessionStart — quando a sessão começa (ótimo pra injetar contexto do projeto).

Cada evento entrega ao seu comando os detalhes do que está acontecendo (qual ferramenta, qual arquivo) em JSON pela entrada padrão. Seu comando lê isso e decide o que fazer.

03 Onde eles moram: settings.json

Hooks vivem no settings.json — no projeto (.claude/settings.json) ou global (~/.claude/settings.json). Você diz o evento, um matcher (em qual ferramenta pega) e o comando:

// .claude/settings.json — formata todo arquivo editado
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "prettier --write $file" }
        ]
      }
    ]
  }
}

Leia de dentro pra fora: no evento PostToolUse, quando a ferramenta for Edit ou Write, rode o comando. Só isso — três camadas: evento → matcher → comando.

04 Um hook que protege

No PreToolUse, seu comando pode bloquear a ação antes dela rodar. Se o comando sai com código 2, o Claude não executa aquilo — e a mensagem que você imprime vira o motivo mostrado a ele. É assim que você barra o perigoso (um rm -rf, um push na main) antes de acontecer.

# bloqueia rm -rf; recebe o comando por stdin (JSON)
grep -q "rm -rf" && { echo "bloqueado: comando destrutivo" >&2; exit 2; }
exit 0

Exit 0 = pode seguir. Exit 2 = barra e devolve o motivo. Simples e poderoso: você transforma "por favor não faça" numa trava real.

05 Ideias pra começar

Você não precisa de muitos. Comece com um hook que te tire uma preocupação repetida da cabeça:

  • Formatar ao salvar — PostToolUse em Edit/Write roda o formatador. Nunca mais código torto.
  • Avisar quando terminar — Stop dispara uma notificação ou um som.
  • Registrar comandos — PreToolUse em Bash grava num log tudo que foi rodado.
  • Injetar as regras do projeto — SessionStart imprime seu CLAUDE.md, e o Claude já começa alinhado.

O melhor hook é o mais chato de fazer na mão. Automatize esse primeiro.

Indo mais fundo

Além do exit code, hooks podem devolver JSON pela saída (uma decisão de aprovar/bloquear com motivo), usar matchers mais finos, e existem outros eventos: SubagentStop, Notification, PreCompact, SessionEnd. Continua nas próximas aulas.

Pratique agora 0/4 feito

Seu primeiro hook: registrar comandos

Faça o Claude gravar cada comando num arquivo — e veja funcionar. Uns 5 minutos, e seguro: o hook só observa (sai com exit 0), não bloqueia nada.

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "cat >> .claude/cmd.log" }
        ]
      }
    ]
  }
}

✓ Pronto: seu primeiro hook rodou sozinho. É exatamente isto — quando isto → roda aquilo. Nas próximas aulas ele fica mais esperto: formatar, avisar, barrar o perigoso.

settings.json
a fundo

Onde os hooks moram, como cada um é montado, e o que o Claude entrega pro seu comando na hora de rodar.

↓ role para estudar

01 Três lugares pros hooks

Um hook pode viver em três arquivos, e todos valem ao mesmo tempo:

  • Global — ~/.claude/settings.json. Vale pra todos os seus projetos.
  • Projeto — .claude/settings.json. Vai pro Git; todo o time herda.
  • Local — .claude/settings.local.json. Fica no .gitignore; é só seu.

Quando há conflito, o mais específico ganha: local sobrepõe projeto, que sobrepõe global. Regra prática: o que é do time vai no .claude/settings.json; o que é gosto seu vai no .local.json.

Teste-se

Qual arquivo é pessoal e NÃO vai pro Git?

02 Anatomia de um hook

Todo hook tem a mesma forma de bonecas russas: o evento guarda uma lista; cada item tem um matcher e uma lista de hooks; cada hook é um comando.

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "./guarda.sh" }
        ]
      }
    ]
  }
}

Pode haver vários hooks no mesmo matcher — eles rodam em ordem. E vários matchers no mesmo evento. Comece com um; a estrutura aguenta crescer.

03 O matcher: onde o hook pega

Para PreToolUse e PostToolUse, o matcher casa com o nome da ferramenta. Três formas:

  • Nome exato — "Bash" pega só o terminal.
  • Alternância — "Edit|Write" pega edição ou escrita de arquivo.
  • Tudo — "*" (ou vazio) pega qualquer ferramenta.

Eventos que não são de ferramenta (como SessionStart ou Stop) não usam matcher de ferramenta — eles simplesmente disparam no momento.

04 O que o hook recebe

Na hora de rodar, o Claude entrega ao seu comando um JSON pela entrada padrão (stdin) com o contexto: qual evento, qual ferramenta, e a entrada dela.

# lê o comando que o Bash ia rodar
cmd=$(jq -r '.tool_input.command')
# lê o arquivo que foi editado
file=$(jq -r '.tool_input.file_path')

Também há a variável $CLAUDE_PROJECT_DIR apontando pra raiz do projeto — útil pra achar seus scripts. Com o stdin + jq, seu hook sabe exatamente o que está prestes a acontecer.

Indo mais fundo

O JSON traz mais campos (session_id, cwd, hook_event_name…). Você raramente precisa de todos — tool_name e tool_input resolvem a maioria dos hooks.

      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "cat >> .claude/edits.log" }
        ]
      }
    ]
  }
}

✓ Você viu o payload real. O matcher Edit|Write e o stdin deixam de ser abstração: é este JSON que o hook lê.

Hooks que
protegem

Transformar "por favor não faça isso" numa trava de verdade: barrar o perigoso antes de acontecer, ou pedir confirmação.

↓ role para estudar

01 A ideia de trava

Uma trava é um hook de PreToolUse: ele roda antes da ferramenta e decide se ela pode seguir. O sinal é o código de saída:

  • exit 0 — tudo bem, pode rodar.
  • exit 2 — barra. O Claude não executa, e o que você imprimiu no stderr vira o motivo.

Repare: a trava não "avisa depois" — ela impede antes. É a diferença entre um extintor e um relatório de incêndio.

Teste-se

Qual código de saída BARRA a ação no PreToolUse?

02 Ler o que vai rodar

Pra decidir, o hook precisa ver o comando. Ele vem no JSON do stdin; você pega com jq e testa:

# .claude/hooks/guarda.sh (PreToolUse, matcher "Bash")
cmd=$(jq -r '.tool_input.command')
if echo "$cmd" | grep -qE 'rm -rf|:(){:'; then
  echo "bloqueado: comando destrutivo" >&2
  exit 2
fi
exit 0

A régua é sua: um grep por padrões perigosos já pega a maioria dos acidentes. Comece simples e vá endurecendo conforme aprende o que te morde.

03 Casos que valem a pena

Três travas que salvam o dia mais cedo do que tarde:

  • Nada de rm -rf em caminhos amplos — o clássico acidente irreversível.
  • Não tocar em segredos — barrar edição de .env, *.pem, chaves.
  • Não empurrar na main — barrar git push direto no branch protegido.

Não precisa prever tudo. Cada susto vira uma linha nova na trava — e ela fica mais esperta com o tempo.

Teste-se

Onde o hook lê o comando que o Bash ia rodar?

04 Pedir confirmação, não só barrar

Nem tudo é preto no branco. Às vezes você não quer proibir, quer ser avisado pra confirmar. Em vez do exit 2, o hook pode devolver um JSON pedindo confirmação:

{ "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "ask",
    "permissionDecisionReason": "confirmar antes de deploy" } }

Assim o Claude pausa e pergunta. Três posturas: allow (libera), deny (barra, como o exit 2) e ask (confirma). Use ask quando o certo depende do contexto.

Indo mais fundo

O exit 2 é o caminho rápido; a saída JSON é a versão fina, com motivo estruturado. Comece pelo exit code — migre pro JSON quando precisar de "ask" ou mensagens mais ricas.

Pratique agora 0/4 feito

Sua primeira trava

Faça um hook barrar um comando de verdade — testando com um token inofensivo, sem chegar perto de um rm -rf.

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command",
            "command": "grep -q BLOQUEIATESTE && { echo 'trava: comando de teste barrado' >&2; exit 2; }; exit 0" }
        ]
      }
    ]
  }
}

✓ Você transformou uma regra em trava real: foi barrado antes de rodar, não é sugestão que o modelo pode ignorar.

Formatar,
notificar, logar

Quatro receitas prontas que tiram trabalho chato da sua frente — cada uma é um hook curto que você cola e esquece.

↓ role para estudar

01 Formatar ao salvar

O clássico. Depois de o Claude editar um arquivo, rode o formatador nele. É PostToolUse com matcher Edit|Write:

{ "hooks": { "PostToolUse": [ {
  "matcher": "Edit|Write",
  "hooks": [ { "type": "command",
    "command": "f=$(jq -r .tool_input.file_path); prettier --write \"$f\"" } ]
} ] } }

O caminho do arquivo vem do stdin. Troque prettier por black, gofmt, o que for do seu projeto. Código sempre limpo, sem você lembrar.

Teste-se

Qual evento roda DEPOIS de um arquivo ser editado?

02 Avisar quando terminar

Você saiu pra pegar um café e o Claude terminou faz tempo. O evento Stop resolve: dispara um aviso quando ele acaba de responder.

# Stop → notificação do sistema
notify-send "Claude" "terminou 👋"   # linux
# osascript -e 'display notification "terminou"'  (mac)

Um afplay/paplay com um som curto também funciona. Pequeno, mas muda seu fluxo: você para de vigiar o terminal.

03 Registrar os comandos

Quer um histórico do que foi rodado? PreToolUse no Bash grava cada comando num arquivo antes de ele executar:

# PreToolUse, matcher "Bash" → append num log
jq -r '.tool_input.command' >> "$CLAUDE_PROJECT_DIR/.claude/cmd.log"
exit 0

Sai com exit 0 (não barra nada, só observa). Depois você tem um diário do que a sessão fez — ótimo pra auditar ou desfazer.

Teste-se

Um hook que só registra (sem barrar) deve sair com:

04 Injetar contexto no começo

O evento SessionStart roda quando a sessão abre, e o que ele imprime vira contexto pro Claude. Use pra ele já começar sabendo do projeto:

# SessionStart → estado atual entra no contexto
echo "Branch: $(git branch --show-current)"
echo "Últimos commits:"; git log --oneline -5

Assim o Claude abre a sessão já ciente do branch, do que mudou, das regras do projeto — sem você colar nada à mão.

Indo mais fundo

UserPromptSubmit faz parecido a cada mensagem sua (injeta contexto ou registra o prompt). Combine os quatro e o ambiente inteiro passa a se cuidar sozinho.

Pratique agora 0/3 feito

Um aviso quando o Claude terminar

Deixe o Claude carimbar um arquivo toda vez que terminar de responder — o gancho do evento Stop. Portável: não depende de desktop.

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "date >> .claude/done.log" }
        ]
      }
    ]
  }
}

✓ Stop virou um sinal seu. Troque o date por um notify-send ou um som e você para de vigiar o terminal.

Hooks em
equipe e CI

O que muda quando os hooks saem da sua máquina: versionar pro time, confiar no determinismo, rodar em pipeline — e a higiene pra não se machucar.

↓ role para estudar

01 Versionar os hooks

Na sua máquina, o hook ajuda você. No repositório, ele ajuda o time inteiro. A diferença é só o arquivo:

  • .claude/settings.json — vai pro Git. Quem clonar o projeto herda os mesmos hooks.
  • .claude/settings.local.json — fica no .gitignore. É seu, não sai da máquina.

Regra: trava de segurança e formatador que todos deveriam ter → versione. Atalho pessoal → local. Assim o padrão do time viaja junto com o código.

Teste-se

Onde colocar um hook para TODO o time herdar?

02 Determinismo: dá pra confiar

Um hook é um comando comum: mesma entrada, mesmo resultado. Ele não "acha" nada, não varia com o humor do modelo. Roda igual na sua máquina, na do colega e no servidor.

É essa previsibilidade que deixa você confiar a segurança a um hook. A trava que barra rm -rf vai barrar sempre — não é uma sugestão que o modelo pode ignorar, é código que sempre executa.

03 Rodando sem ninguém olhando

O Claude Code roda também em modo headless — sem interface, tocado por script (claude -p "…"), dentro de um pipeline de CI. E os hooks continuam valendo.

Ou seja: o formatador, a trava e o log que você montou pra usar no dia a dia também protegem a automação. O mesmo padrão vale pro humano e pro robô — você configura uma vez.

Teste-se

No modo headless (CI), os hooks configurados:

04 Higiene: hook é poder

Um hook roda comandos automaticamente — então trate com respeito:

  • Rápido — ele entra no caminho de cada ação; um hook lento trava seu fluxo.
  • Idempotente — rodar duas vezes não pode quebrar nada.
  • Sem vazar segredo — não imprima tokens/chaves; o output pode ir pro contexto ou pro log.
  • Falhe com clareza — mensagem no stderr dizendo o porquê.

Comece com um hook simples e seguro. O objetivo não é automatizar tudo — é tirar da sua cabeça as poucas coisas que você não devia precisar lembrar.

Indo mais fundo

Dá pra dar timeout aos comandos e escolher a ordem de execução quando há vários hooks no mesmo matcher. Mas isso é afinação — a base é: rápido, idempotente, silencioso sobre segredos.

Pratique agora 0/4 feito

Versione pro time · capstone

Consolide os hooks que você montou e faça o padrão viajar com o repositório — quem clonar herda tudo.

# suas travas/formatadores vão no arquivo versionado
git add .claude/settings.json
git status
# o que é pessoal (só seu) fica de fora do Git:
echo ".claude/settings.local.json" >> .gitignore

✓ O padrão do time agora viaja junto com o código — rápido, idempotente, sem vazar segredo. Vale pro humano e pro CI. Fim da trilha. 🎉
